# Feature Builder - AI 辅助功能开发流程

## 🎯 技能描述

本技能提供标准化的 AI 辅助功能开发流程，确保高质量、高效率地完成功能开发。

**触发条件**：
- 用户请求实现新功能（如"帮我实现登录功能"）
- 用户要求构建模块或组件
- 用户需要完成某个开发任务
- 用户提到"开发"、"实现"、"构建"等功能性需求

**核心原则**：
1. ✅ 先检索文档，再生成代码
2. ✅ 分步骤实现，及时审查
3. ✅ AI 自我审查代码质量
4. ✅ 测试驱动开发
5. ✅ 及时更新文档

---

## 📋 标准流程

### Step 1: 需求理解（Requirements Understanding）

#### 1.1 检索需求文档
```
让我先查看需求文档了解具体要求。
[检索 requirements/ 目录下的相关文档]
```

**必须查看**：
- `requirements/README.md` - 了解文档结构
- `requirements/02-functional-requirements.md` - 功能需求规格
- `requirements/01-project-background.md` - 项目背景（如需要）

#### 1.2 提取需求要点
返回以下信息：
- 功能需求列表（带需求 ID，如 FR-USER-001）
- 优先级（P0/P1/P2/P3）
- 验收标准
- 相关约束条件

#### 1.3 确认需求理解
```
根据需求文档，[功能名称] 需要实现：
- ✅ [需求 1]
- ✅ [需求 2]
- ✅ [需求 3]

请问还有其他需求或特殊要求吗？
```

**示例输出**：
```
根据需求文档 2.1 节，用户登录功能需要实现：
- FR-USER-001: 支持邮箱格式验证
- FR-USER-002: 支持密码强度校验（至少 8 位，包含字母数字）
- FR-USER-003: 注册时发送验证邮件
- FR-USER-004: 支持邮箱验证码激活（5 分钟有效期）
- FR-USER-006: 支持记住登录状态（7 天）

优先级：P0
验收标准：
- 邮箱格式错误时即时提示
- 密码强度不足时显示强度指示器
- 验证码 5 分钟内有效

请问还有其他需求吗？
```

### Step 2: 技术方案设计（Technical Design）

#### 2.1 检索设计文档
```
让我查看设计文档了解技术架构。
[检索 DESIGN.md 相关章节]
```

**必须查看**：
- `DESIGN.md` - 系统架构、数据库设计、API 设计
- 相关技术栈和框架说明

#### 2.2 提出技术方案
返回以下内容：
1. **架构设计**：模块划分、层次结构
2. **技术选型**：使用的库、框架
3. **数据结构**：数据库表、实体类
4. **接口设计**：API 端点、请求/响应格式
5. **实现步骤**：分步实现计划

**示例输出**：
```
根据设计文档，建议实现方案：

架构设计：
- Controller 层：AuthController（处理 HTTP 请求）
- Service 层：AuthService（业务逻辑）
- Repository 层：UserRepository（数据访问）

技术选型：
- Spring Security 6+ JWT 认证
- 密码加密：bcrypt
- Token 存储：HttpOnly Cookie
- 验证框架：Hibernate Validator

数据结构：
- User 实体（参考 DESIGN.md 第 2.1 节）
- 新增 verification_codes 表（存储验证码）

API 设计：
- POST /api/auth/register - 用户注册
- POST /api/auth/verify - 验证邮箱
- POST /api/auth/login - 用户登录

实现步骤：
1. 创建 User 实体类
2. 实现 UserRepository
3. 创建注册请求 DTO
4. 实现 AuthService
5. 实现 AuthController
6. 前端实现（如需要）

同意这个方案吗？需要修改吗？
```

#### 2.3 讨论技术细节
- 回答用户关于技术选型的问题
- 根据用户反馈调整方案
- 确认实现优先级（如先实现核心功能，增强功能后续迭代）

### Step 3: 代码生成（Code Generation）

#### 3.1 分模块生成原则
**禁止**一次性生成所有代码，必须：
1. 按模块分步生成
2. 每步完成后等待审查
3. 根据反馈修改后再继续

#### 3.2 后端实现顺序
```
Step 3.1: 创建实体类
[生成 Entity.java]
请审查代码，特别注意：
- 字段注解是否正确
- 数据类型是否合适
- 是否有 @JsonIgnore 等安全注解

需要修改吗？

Step 3.2: 创建 Repository
[生成 Repository.java]

Step 3.3: 创建 DTO/Request/Response
[生成相关类]

Step 3.4: 实现 Service 层
[生成 Service.java]

Step 3.5: 实现 Controller 层
[生成 Controller.java]

Step 3.6: 配置和工具类
[生成 Config.java, Util.java]
```

#### 3.3 前端实现顺序（如需要）
```
Step 3.1: 类型定义
[生成 types.ts]

Step 3.2: API 服务
[生成 api.ts]

Step 3.3: 状态管理
[生成 store.ts]

Step 3.4: 组件实现
[生成 Component.tsx]

Step 3.5: 样式文件
[生成 Component.module.css]
```

#### 3.4 代码审查要点
每生成一段代码后，必须：
1. **自我审查**：让 AI 检查代码问题
2. **提示审查点**：告诉用户需要关注什么
3. **等待确认**：用户确认后再继续

**自我审查清单**：
```
代码审查：
- [ ] 符合需求文档所有要求
- [ ] 符合设计文档架构
- [ ] 无安全漏洞（SQL 注入、XSS 等）
- [ ] 有完整的错误处理
- [ ] 有日志记录
- [ ] 命名规范一致
- [ ] 代码注释清晰

发现的问题：
❌ [问题 1]
❌ [问题 2]
✅ [做得好的地方]

建议修复这些问题。
```

---

### Step 4: 代码审查（Code Review）

#### 4.1 AI 自我审查
```
让我审查刚才生成的代码，检查：
1. 安全性：SQL 注入、XSS、CSRF 防护
2. 性能：数据库查询、缓存使用
3. 错误处理：异常捕获、日志记录
4. 代码质量：命名规范、代码结构

发现以下问题：
❌ 问题 1：...
❌ 问题 2：...
✅ 优点：...

需要我修复这些问题吗？
```

#### 4.2 安全问题检查
**必须检查**：
- SQL 注入防护（参数化查询）
- XSS 防护（输入验证、输出编码）
- CSRF 防护（Token 验证）
- 敏感数据加密（密码、Token）
- 权限验证（登录检查、角色验证）

#### 4.3 修复问题
根据审查结果修复代码，并说明：
- 修复了什么问题
- 如何修复的
- 修复后的影响

#### 5.1 查看测试需求
```

```

#### 5.2 生成测试用例
**单元测试**：
```

- [测试场景 1]
- [测试场景 2]
- [边界情况]

✅ 异常场景（X 个用例）
✅ 边界情况（X 个用例）

```

**集成测试**（如需要）：
```
生成集成测试：
[生成 IntegrationTest.java]
使用 Testcontainers 进行真实环境测试
```

#### 5.3 运行测试
```
[运行测试命令]
测试结果：
✅ 通过：X 个
❌ 失败：X 个

修复失败的测试：

```

#### 5.4 测试覆盖率检查
```

- 行覆盖率：XX%（目标 > 80%）
- 分支覆盖率：XX%（目标 > 70%）

覆盖率 [达标/不达标]，[需要/无需] 补充测试。
```

## ⚠️ 注意事项

### ✅ DO（必须做）
1. ✅ **必须先检索需求文档** - 确保理解完整需求
2. ✅ **必须提出技术方案并确认** - 避免方向错误
3. ✅ **必须分步骤实现** - 小步快跑，及时审查
4. ✅ **必须 AI 自我审查** - 主动发现并修复问题
5. ✅ **必须生成完整测试** - 确保功能正确
6. ✅ **必须更新文档** - 保持文档与代码同步

### ❌ DON'T（禁止做）
1. ❌ 不检索文档直接写代码
2. ❌ 一次性生成所有代码
3. ❌ 不进行自我审查
4. ❌ 忽略测试或测试不完整
5. ❌ 忘记更新文档
6. ❌ 跳过技术方案讨论
