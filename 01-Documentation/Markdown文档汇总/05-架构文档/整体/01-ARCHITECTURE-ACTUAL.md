## 架构偏差分析

### 1️⃣ 多端应用 (高优先级)

**原设计**:
```
├─ Web (React 18)
├─ Mobile (Flutter 3.x)
└─ WeChat MiniProgram (Taro 4.x)
```

**当前实现**:
```
├─ Web (React 18) ✅
├─ Mobile (Flutter) ❌ 未实现
└─ WeChat MiniProgram (Taro) ❌ 未实现
```

**原因**: MVP优先选择Web应用，移动端计划在Phase 2

**影响**: 无法支持App和小程序用户

**行动**: 参考 02-ROADMAP.md 的移动端规划

---

### 2️⃣ 高级基础设施 (中等优先级)

**原设计包含**:
- Milvus (向量知识库)
- MinIO/OSS (文件存储)
- Meilisearch (搜索引擎)
- ShardingSphere (分库分表)
- RocketMQ (消息队列)

**当前实现**:
- ❌ Milvus - 无法启用向量搜索
- ❌ MinIO/OSS - 无文件存储
- ❌ Meilisearch - 无高级搜索
- ❌ ShardingSphere - 无分库分表
- ❌ RocketMQ - 仅有Redis Streams

**原因**: 当前业务量不需要，计划支持规模化后再加入

**影响**: 
- 无法支持大规模知识库向量检索
- 文件存储需要自行实现或使用云存储
- 跨服务数据一致性依赖应用层实现

**行动**: 参考 02-ROADMAP.md 的基础设施规划

---

### 3️⃣ 可观测性栈 (中等优先级)

**原设计包含**:
- ELK Stack (日志收集)
- SkyWalking/Jaeger (链路追踪)
- Prometheus/Grafana (监控)

**当前实现**:
- ❌ 未配置任何可观测性

**原因**: MVP阶段重点在功能实现，监控规划在生产化阶段

**影响**: 生产问题排查困难，性能瓶颈难以定位

**行动**: 参考 02-ROADMAP.md 的可观测性规划

---

### 4️⃣ AI服务 - LangChain未集成 (低优先级)

**原设计**:
```
Prompt工程 + LangChain + 多模型支持 (文心一言/GLM-4/通义千问)
```

**当前实现**:
```
直接DashScope调用 + 仅通义千问支持
```

**原因**: 当前业务只需单一模型，LangChain在需要Prompt框架时再集成

**影响**: 无统一的Prompt管理，模型切换需要改代码

**行动**: 参考 02-ROADMAP.md 的AI服务升级规划

---

### 5️⃣ 服务治理 - 事务管理缺失 (低优先级)

**原设计**: Seata AT模式分布式事务

**当前实现**: 本地事务 + 应用层补偿

**原因**: 当前业务复杂度不需要，后续考虑

**影响**: 跨服务操作无原子性保证

**行动**: 业务需要时按需集成Seata
