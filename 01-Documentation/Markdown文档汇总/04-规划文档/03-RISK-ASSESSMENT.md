# 风险评估与应对措施

📌 **文档概述**  
本文档系统地分析全自动家教备课平台项目面临的关键风险，包括技术风险、运营风险、人力资源风险等，为每个风险提供应对策略和责任人。通过风险管理能够提前识别问题、降低项目失败风险、提高交付质量。

⏱️ **阅读时间**  
12-15 分钟

🎯 **适用场景**  
- 项目启动时的风险识别和规划
- 周会风险评估和状态更新
- 应急预案制定
- 团队风险意识培养

---

## 风险评估框架

### 风险等级定义

| 等级 | 概率 × 影响 | 描述 | 应对方式 |
|------|----------|------|--------|
| **极高** | 高概率 × 高影响 | 项目交付受威胁 | 制定详细应急预案，持续监控 |
| **高** | 中高概率 × 高影响或高概率 × 中影响 | 交付时间或质量风险 | 预防为主，专项资源投入 |
| **中** | 中概率 × 中影响或低概率 × 高影响 | 局部影响 | 监控和应对计划 |
| **低** | 低概率 × 低/中影响 | 可接受 | 常规管理 |

## 高风险（需要重点关注）

### 1. AI 生成内容准确性低 ⚠️ 极高风险

| 项目 | 内容 |
|------|------|
| **风险描述** | 生成的教案在知识准确性、教学逻辑或题目难度上存在较多问题，影响用户信任 |
| **概率** | 高 | 
| **影响** | 高（直接影响核心功能可用性和用户满意度） |
| **风险值** | **极高** |

#### 应对措施
1. **知识库建设**
   - 建立高质量的知识点库（> 500 个知识点）
   - 每个知识点包含准确的定义、公式、常见误区
   - 定期由专家审核和更新
   - 导入到 Milvus 向量库，支持 RAG 增强

2. **Prompt 优化**
   - 采用 Few-shot 学习，准备 50+ 优质案例
   - 使用 Chain-of-Thought 激发模型推理能力
   - 定期进行 A/B 测试，迭代 Prompt
   - 准备多个版本的 Prompt，针对不同教学场景

3. **内容审核机制**
   - 实现自动化内容审核
     - 事实检查：知识点准确性校验
     - 格式检查：教案结构完整性
     - 敏感词检查：内容安全性
   - 建立人工审核队伍
     - 初期 Phase 1-2 的所有生成内容进行人工审核
     - 反馈机制：不准确的内容收集，用于优化 Prompt
   - 用户反馈闭环
     - 用户可标注错误，系统收集错误数据
     - 定期分析错误模式，优化 Prompt 和知识库

4. **质量评分系统**
   - 多维度评分模型
     - 知识准确度（0-100）
     - 教学逻辑性（0-100）
     - 难度匹配度（0-100）
     - 完整性（0-100）
   - 低分内容：不展示或标记为"需要审核"
   - 积累评分数据，训练评分模型

5. **降级和备选方案**
   - 当准确性低于 80% 时，自动降低 Token 消耗的措施
   - 优先使用更强的模型（qwen-max）
   - 多次生成取最好的结果（Top-K 投票）

#### 监控指标
- 每周计算生成内容准确率（目标 > 80%，Phase 3 达到 > 85%）
- 用户反馈中的错误率（目标 < 5%）
- 人工审核中的驳回率

#### 负责人
**AI 工程师 + 产品经理**

### 2. 大模型 API 成本过高 ⚠️ 极高风险

| 项目 | 内容 |
|------|------|
| **风险描述** | 大模型 API 调用成本高于预期，导致边际成本过高，商业模式不可行 |
| **概率** | 高 |
| **影响** | 高（影响项目经济可行性） |
| **风险值** | **极高** |

#### 应对措施
1. **多级缓存策略（最优）**
   - **Redis 分布式缓存**
     - 缓存 Key: `lesson:{user_id}:{student_id}:{topic}:{mode}`
     - 存储完整生成结果
     - TTL: 7 天（用户可能会生成多份相同内容）
     - 预期命中率: 60%（相同学生的相同知识点）
   
   - **本地 Caffeine 缓存**
     - 缓存热门知识点的讲解模板（模板化内容）
     - 缓存常用 Prompt 和 Few-shot 示例
     - TTL: 24 小时
     - 预热: 每天凌晨加载
     - 减少 Token 消耗
   
   - **结果缓存**
     - 如果输入完全相同（计算 MD5 指纹），直接返回缓存
     - 不需要再调用大模型

   **成本影响**: 缓存命中 60% 时，API 成本降低 60%

2. **Token 消耗优化**
   - Prompt 精简
     - 移除不必要的解释和示例
     - 使用简洁的格式描述
   - Few-shot 动态选择
     - 只包含最相关的 3 个示例，而不是 5 个
   - 输出格式优化
     - 使用更紧凑的结构化格式（减少 Token）
   - 预计节省: 15-20% Token 消耗

3. **模型层级化策略**
   - **快速问答** (duration < 20 min) → qwen-turbo（最便宜）
   - **标准备课** → qwen-plus（性价比好）
   - **深入讲解** → qwen-max（最强）
   - **默认失败** → ChatGLM 本地模型（0 成本）
   - 预计节省: 30% 成本（多用便宜模型）

4. **本地模型备选方案**
   - **ChatGLM 3-6B 本地部署**
     - 在服务器本地部署开源大模型
     - 成本: 计算成本（GPU），0 API 调用成本
     - 准确率: 比云模型略低，但对简单任务足够
     - 使用场景: 快速问答、简单讲解、备选方案
   - **Fallback 逻辑**
     ```
     如果云模型 API 失败或成本过高:
       ├─ 尝试 qwen-turbo（快速便宜版本）
       ├─ 如果继续失败 → 尝试 ChatGLM 本地
       └─ 如果仍然失败 → 返回友好错误提示
     ```

5. **按需计费模型**
   - 免费额度: 每个用户每月 3 次备课（吸引用户）
   - 超出额度: 按次付费或升级为订阅
   - 预期: 大多数用户留在免费额度内，成本可控

6. **成本预算控制**
   - 设定每日 API 调用成本上限 (Budget Alert)
   - 超出预算时触发告警
   - 临时禁用最贵的模型（qwen-max），使用次级模型
   - Nacos 配置中心动态调整模型选择

#### 监控指标
- 每次备课生成的平均 Token 消耗
- 平均 API 成本 (¥/次)
- 缓存命中率（目标 60%+）
- 月度 API 总成本与预算对比

#### 负责人
**技术负责人 + AI 工程师**

### 3. 核心开发人员离职 ⚠️ 高风险

| 项目 | 内容 |
|------|------|
| **风险描述** | 关键技术人员离职，导致项目知识丧失、进度延缓、技术债务累积 |
| **概率** | 中 |
| **影响** | 高（关键模块无人维护） |
| **风险值** | **高** |

#### 应对措施
1. **知识文档化**
   - 架构决策记录（ADR）
     - 每个重要技术决策都要记录为什么这样选择
     - 存储在 docs/adr 目录
     - 例: `001-use-taro-for-miniprogram.md`
   - 代码注释和 Javadoc
     - 复杂逻辑必须有详细注释
     - 重要算法的数学推导要写清楚
   - 部署和维护文档
     - 每个服务的启动、配置、故障排查文档
     - 定期更新

2. **知识分享**
   - **每周 Tech Talk**（周五下午 2 点）
     - 分享技术踩坑和最佳实践
     - 轮流分享，让每个人都有机会讲
     - 录制视频，保存到知识库
   - **Pair Programming**
     - 关键模块采用结对编程
     - 2 个人都懂该模块，任何人离职都有备份
   - **代码审查**
     - 严格的 Code Review，传播知识

3. **关键模块双人备份**
   - AI 备课模块（FastAPI + LangChain）
     - 主负责人: AI 工程师 A
     - 备份负责人: AI 工程师 B
   - 数据同步模块（WebSocket）
     - 主负责人: 后端开发 X
     - 备份负责人: 后端开发 Y
   - 前端架构
     - 主负责人: 前端开发 P
     - 备份负责人: 前端开发 Q

4. **完善 CI/CD 减少人工依赖**
   - 自动化部署
     - 提交代码 → 自动构建 → 自动测试 → 自动部署
     - 不依赖某个人的手工操作
   - 自动化测试
     - 单元测试覆盖率 > 70%，发现 bug 及时
   - 自动化监控
     - 问题自动告警，不需要人盯着

#### 监控指标
- 文档覆盖率（目标 100% 关键模块）
- 每个模块至少 2 人懂该技术
- 知识分享频率（每周 1 次）

#### 负责人
**项目经理 + 技术负责人**

## 中风险（需要定期关注）

### 4. 多端同步复杂度高 ⚠️ 中高风险

| 项目 | 内容 |
|------|------|
| **风险描述** | Web / App / 小程序 三端数据同步复杂，冲突处理困难，易导致数据不一致 |
| **概率** | 中 |
| **影响** | 高（影响用户体验和数据安全） |
| **风险值** | **中高** |

#### 应对措施
1. **采用 Taro 一套代码多端**
   - 小程序使用 Taro 框架，与 Web 端共用 80% 代码
   - 减少维护工作量和差异导致的 bug
   - 预计减少开发工作量 30%

2. **冲突检测和字段级合并算法**
   - 参考文档 `01-MODULE-DETAILS.md` 的 "冲突检测与合并算法" 部分
   - 核心思想: 三向合并（base, local, remote）
   - 字段级粒度，而不是整个记录级

3. **充分的测试覆盖**
   - 同步测试用例 > 50 个
     - 网络中断恢复
     - 并发修改同一字段
     - 离线修改后在线冲突
   - E2E 测试覆盖所有端的同步流程
   - 性能测试: 10 并发用户同步，无数据丢失

4. **预留 2 周专项优化**
   - Phase 3 Week 7 专项用于同步调试和优化
   - 团队联合测试（Web + App + 小程序）
   - 发现问题及时修复

#### 监控指标
- 数据一致性检查（定期扫描数据库）
- 同步失败率（目标 < 0.1%）
- 冲突发生率和解决率

#### 负责人
**后端负责人 + 前端负责人**

### 5. 数据安全问题 ⚠️ 中风险

| 项目 | 内容 |
|------|------|
| **风险描述** | 学生个人信息、教师账户被泄露或被攻击，导致法律责任和用户信任丧失 |
| **概率** | 低 |
| **影响** | 高（涉及用户隐私和法律责任） |
| **风险值** | **中** |

#### 应对措施
1. **安全代码审查（SAST）**
   - 使用 SonarQube 自动检测代码漏洞
   - 检查项:
     - SQL 注入漏洞
     - XSS 漏洞
     - 敏感信息硬编码
     - 加密算法错误使用
   - 目标: 0 高危漏洞，P0 critical issues 必须修复

2. **渗透测试**
   - Phase 3 Week 10 进行一次正式渗透测试
   - 测试项:
     - 认证绕过（能否伪造 Token）
     - 权限提升（普通用户能否访问他人数据）
     - 数据泄露（数据库备份是否加密）
     - 接口安全（API 是否有速率限制）
   - 修复所有发现的漏洞

3. **数据加密**
   - **传输加密**: HTTPS/TLS 1.3，所有 API 必须走 HTTPS
   - **存储加密**: 
     - 敏感字段（身份证号、学生名字等）使用 AES-256-GCM 加密存储
     - 密钥管理: 使用 Nacos 配置中心管理，定期轮换
     - 密钥不在代码中硬编码
   - **密码加密**: bcrypt（salt rounds: 12），不存储明文密码

4. **定期安全审计**
   - 每 3 个月进行一次内部安全审计
   - 检查项:
     - 日志完整性（有无异常访问）
     - 备份完整性（备份是否可恢复）
     - 访问控制（权限划分是否合理）
   - 生成审计报告，跟踪修复进度

5. **隐私合规**
   - GDPR 合规（如果有欧洲用户）
   - 中国《个人信息保护法》合规
   - 用户数据导出功能（用户权利）
   - 数据删除功能（用户有权要求删除）

#### 监控指标
- SonarQube 高危漏洞数（目标: 0）
- 安全漏洞修复时间（目标: 48 小时内修复 P0）
- 渗透测试结果

#### 负责人
**安全工程师**

### 6. 应用商店审核不通过 ⚠️ 中风险

| 项目 | 内容 |
|------|------|
| **风险描述** | Flutter App 或小程序因内容、功能等原因被应用商店（Apple、Android、微信等）拒审，影响上线 |
| **概率** | 中 |
| **影响** | 中（延缓上线时间） |
| **风险值** | **中** |

#### 应对措施
1. **提前了解审核规则**
   - 阅读 App Store、Google Play、微信小程序的审核指南
   - 了解常见拒审原因
   - 咨询行业经验（找有过上线经验的团队咨询）

2. **准备多套方案**
   - 敏感功能（如分享、支付）准备多个实现方案
   - 内容敏感词库，自动过滤
   - 隐私政策、用户协议准备完善版本

3. **预留审核时间缓冲**
   - 实际上线日期 = 目标日期 - 2 周
   - 预留时间应对审核拒绝和重新提交

4. **建立申诉渠道**
   - 如果被拒审，分析拒审原因
   - 针对性修改和改进
   - 准备申诉材料，解释为什么应该通过

#### 监控指标
- 审核周期（平均几天通过）
- 拒审率和常见拒审原因

#### 负责人
**产品负责人 + 运营经理**

### 7. 需求变更频繁 ⚠️ 中风险

| 项目 | 内容 |
|------|------|
| **风险描述** | 产品需求频繁变更，导致开发计划混乱、团队工作效率下降、交付时间延迟 |
| **概率** | 中 |
| **影响** | 中（影响项目时间表） |
| **风险值** | **中** |

#### 应对措施
1. **敏捷开发，短迭代周期**
   - 采用 Scrum 框架，2 周一个 Sprint
   - 每个 Sprint 的需求在 Sprint 开始前确定，中途不变更
   - 新需求进入 Backlog，等下个 Sprint 再处理

2. **需求变更控制流程**
   ```
   需求变更请求
     ↓
   产品经理评估影响（时间成本、优先级）
     ↓
   技术负责人评估开发成本
     ↓
   决策: 接受 / 推迟 / 拒绝
     ↓
   如果接受: 调整计划和资源分配
   ```

3. **优先级管理（MoSCoW 法则）**
   - **Must** (必须有): Phase 1 必须完成的功能
   - **Should** (应该有): Phase 2 期望有的功能
   - **Could** (可以有): Phase 3 可选优化
   - **Won't** (暂不做): 放入 Phase 4 或砍掉
   - 清晰的优先级避免"什么都想要"的局面

4. **预留 20% 缓冲时间**
   - 每个 Sprint 计划时，预留 20% 的时间用于应对临时问题和需求变更
   - 例: 2 周 = 10 个工作日，实际计划 8 个工作日的任务

#### 监控指标
- 需求变更频率（每个 Sprint 新增需求数）
- 变更导致的时间延迟

#### 负责人
**产品经理 + 项目经理**

## 低风险（定期回顾）

### 8. 用户学习成本高 ⚠️ 低风险

**应对措施:**
- 交互式新手引导 (Shepherd.js)
- 丰富的示例数据，让用户快速上手
- 视频教程（快速入门 5 分钟，进阶 20 分钟）
- 在线客服支持（初期 PM 兼顾，后期招客服）
- **目标**: 新用户首次备课完成时间 < 10 分钟

---

### 9. 竞品抢先发布 ⚠️ 低风险

**应对措施:**
- 加快 MVP 发布速度（4 周内）
- 聚焦差异化功能（AI 备课的深度）
- 快速迭代响应市场
- **防守策略**: 即使晚发布，也要差异化更明显

---

### 10. 技术选型失误 ⚠️ 低风险

**应对措施:**
- 技术预研和 POC 验证（新技术选型前做小规模验证）
- 技术决策文档化 (ADR)
- 保持架构可扩展性（方便技术栈升级）

---

## 风险监控机制

### 每周风险 Review

**时间**: 每周一下午 3 点，项目管理会

**参与者**: 产品经理、技术负责人、各模块负责人

**议程**:
1. 上周风险状态更新（5 分钟）
2. 新发现的风险（10 分钟）
3. 高风险应对措施评审（15 分钟）
4. 决策和行动项（10 分钟）

**输出**: 
- 风险 Dashboard 更新（Jira 中）
- 行动项和负责人确认

```
│ ├─ 审核不通过 (预防中)                       │
│ └─ 需求变更 (应对中)                         │
│                                              │

└─────────────────────────────────────────────┘
```

### 预警机制

**风险升级条件**:
- 高风险的监控指标恶化（例: 准确率从 82% 降至 75%）
- 中风险发生，升级为高风险
- 新发现的风险，当场评估等级

**升级通知**:
- 使用 PagerDuty 自动告警
- Slack 频道通知全体
- 项目经理跟进处理
